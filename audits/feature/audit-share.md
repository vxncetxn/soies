## The three observed issues

### 1. Copy toast moves the sheet

**Confidence:** Medium

`ShareSheet.tsx:408-420, 525-531` conditionally mounts the toast inside a sheet using the live `"content"` detent. The Copy anchor computes to 10% minus 64 px, which is negative on normal phones; Download remains inside bounds. This is a concrete Copy-only difference, but static code cannot prove it is the sole trigger because both actions mount the same toast subtree.

**Causal chain**

1. Copy succeeds → toast node mounts
2. First anchor overflows left; Copy also completes expensive Base64 clipboard work
3. The dynamic content detent can remeasure and retarget
4. Toast unmounts → the sheet returns to its previous detent

**Recommended fix:** Keep a fixed-height toast lane mounted at all times, clamp the first/last pill within sheet bounds, and animate only opacity/transform. A fixed numeric detent or `animateContentHeight=false` is a secondary guard, not the primary fix.

### 2. Every share action freezes

**Confidence:** High

`ShareCaptureHost.tsx:109-119` mounts a fresh 1080 × 1920 logical-point React tree for every action. A Print also remounts and decodes its photo. Then `captureRef` rasterizes and PNG-compresses it. The two animation frames in `ShareSheet.tsx:168-175` only let the spinner paint before this blocking work; they cannot keep it animating.

**Causal chain**

1. Tap → mount full export tree
2. Print photo decode + layout
3. Native snapshot + lossless PNG compression
4. Unmount; repeat the whole path on the next tap

Copy adds a second hotspot at `destinations.ts:25-32`: the full PNG is read into a Base64 JS string and sent to the native clipboard.

**Recommended fix:** Render at target pixels divided by `PixelRatio`, use JPEG for opaque canvas exports and PNG only for stickers, cache completed captures by artefact/background/variant for the sheet session, and release temporary files. If first-tap latency is still visible, pre-render the selected canonical composition after the sheet settles.

### 3. Print preview is not 1:1

**Confidence:** High

`ShareArtefactCard.tsx:29-34` uses `screenWidth - 80` as the base width for both Paper and Print. The real Print geometry in `ArtefactWrapper.tsx:25-28` and `EditablePrint.tsx:112-118` derives a narrower width from the common paper height: `(53 / 86) × paperHeight`. Because `pt-8`, `gap-4`, font size, and line height are fixed at base layout size before scaling, scaling from the wrong base moves the caption relative to the image.

**Recommended fix:** Extract one shared artefact-layout function and shared Print caption tokens. Reuse them in `ArtefactWrapper`, `EditablePrint`, `Print`, and `ShareArtefactCard`. Remove the preview-only minimum brand font and fixed shadow values if true WYSIWYG export is required.

> **Note:** Editable Print explicitly uses 16 px text with 22.4 px line height and zero padding/margin, while display Print relies on utility defaults. This should be one caption primitive or token set.

## Recommended remediation order

1. Make capture single-flight, timeout-safe, image-readiness-aware, and cleanup-safe.
2. Right-size the logical render surface, use JPEG for opaque canvases, and profile first-tap capture on representative iOS and Android devices.
3. Centralize Paper/Print geometry and caption typography; add snapshot dimensions and visual comparison tests.
4. Stabilize the toast lane and the controlled sheet detent.
5. Remove dead dependencies/assets, format, and run the full device matrix before release.

## Verification gate

Record spinner FPS and action latency for Paper and Print, repeat each action 20 times while watching memory/temp storage, verify Ink is present, test image-load failure and rapid double taps, reopen the sheet on a different page, and compare actual/preview/export overlays at 50% opacity.
