# Global application adversarial audit 01

**Audit date:** 2026-07-18

**Baseline:** <code>f18fbfe</code> on <code>main</code>; the worktree was clean when review began

**Scope:** Expo SDK 57 application code, configuration, dependencies, SQLite layer, storage/media flows, routing, animations, custom iOS/Android text and Ink implementations, vendored native code, widgets, share/create flows, tests, and release tooling

## Executive verdict

The app passes its current static quality gate and both production JavaScript exports complete, but it is not yet release-hardened against the requested performance and failure-handling bar. The largest risks are architectural rather than lint-level: Home eagerly loads and retains the whole journal; a non-virtualized pager multiplies full-screen portal and blur subtrees; iOS Paper typing performs repeated full TextKit layouts per key; Ink has unbounded live history and a native request path that can leave Promises pending; Android does not implement the bounded-text contract presented by the UI; and several visible actions are intentionally enabled no-ops.

No evidence-backed Critical issue was found. Thirteen High findings should nevertheless be treated as release blockers because they can produce unbounded memory/work, apparent data loss, broken primary actions, or platform-divergent saved content.

## Severity rubric

- **Critical:** likely catastrophic data loss, repeatable crash, or inability to use the app with no practical recovery.
- **High:** release-blocking risk of major user-visible failure, unbounded resource growth, platform contract violation, or severe degradation at realistic scale.
- **Medium:** meaningful reliability, latency, permission, bloat, or maintenance issue whose impact is bounded or depends on a narrower trigger.
- **Low:** cleanup or future-risk issue with little present user impact.

## Recommended remediation sequence

## Device validation gate

The remediation pass added clean native generation, an iOS Release simulator build, and a four-ABI Android Release assembly to the original source/config, Node/static, Expo, and production-bundle validation. It did **not** run the rebuilt application on physical devices or collect simulator/device instrumentation, so animation smoothness, launch time, native heap behavior, permission UI, process-death restoration, and platform crash-freedom remain unproven.

Before release, record at minimum:

- clean-install, cold, and warm time-to-interactive on a lower-tier Android device and the oldest supported iPhone;
- JS/UI FPS and dropped frames while opening/closing Calendar and Focus, paging 100 entries, and navigating uncached dates;
- JS heap, native heap/RSS, and bitmap/PencilKit growth during long drawing, undo/replay, rotation, and repeated Save;
- p50/p95 key-to-paint latency near Paper capacity;
- 20-cycle create/cancel/save/share tests with temp-directory and durable-file accounting;
- injected database migration, corrupt JSON, missing image/Ink, permission denial, native callback loss, and process-death recovery.
